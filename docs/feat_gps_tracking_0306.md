# GPS轨迹记录功能优化方案

**版本**: 1.0
**日期**: 2026-03-06
**状态**: 方案设计

---

## 1. 现状分析

### 1.1 当前实现

当前`ActiveWorkoutScreen`存在的问题：
- 仅显示数字统计（距离、时长、卡路里）
- 无地图可视化展示
- GPS点已采集但未展示给用户
- 用户无法直观看到运动轨迹

### 1.2 数据模型现状

已有`GPSPoint`模型包含：
- `latitude` / `longitude` - 经纬度
- `timestamp` - 时间戳
- `altitude` - 海拔（可选）
- `speed` - 速度（可选）

运动记录已支持存储`gpsPoints`列表，保存到数据库的JSON字段中。

## 2. 竞品调研

### 2.1 主流健身APP地图功能

| APP | 实时地图 | 轨迹缩略图 | 离线地图 | 特色 |
|-----|----------|------------|----------|------|
| **Keep** | 全屏地图 | 列表卡片小图 | 不支持 | 轨迹渐变色 |
| **悦跑圈** | 全屏地图 | 列表卡片小图 | 支持缓存 | 热力图轨迹 |
| **Strava** | 全屏地图 | 详情页大图 | 支持 | 全球路线搜索 |
| **Nike Run** | 半屏地图 | 路线概览图 | 部分支持 | 动画轨迹 |

### 2.2 关键交互模式

1. **实时运动中**：
   - 半屏地图 + 半屏数据（主流方案）
   - 全屏数据 + 小地图浮窗
   - 全屏地图 + 底部数据栏

2. **运动记录列表**：
   - 缩略图卡片（左图右文）
   - 纯文字 + 路线距离

3. **运动详情**：
   - 全屏可交互地图
   - 路线统计摘要

#### 技术方案

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 地图SDK | `flutter_map` + `latlong2` | 开源无需API Key |
| 地图瓦片 | OpenStreetMap | 免费开源 |
| 轨迹渲染 | Polyline + 渐变色 | flutter_map自带 |
| 定位追踪 | `geolocator` | 现有方案保留 |
| 缩略图 | 地图快照（手动实现） | 用RepaintBoundary截图 |

#### 技术实现要点

1. **实时地图**：
   - 使用`flutter_map`替换纯数据展示
   - `PolylineLayer`绘制轨迹线
   - `LocationMarker`显示当前位置
   - 每5秒更新一次地图中心点

2. **缩略图生成**：
   - 在详情页使用`RepaintBoundary`生成快照
   - 保存为Base64或文件到本地
   - 列表页直接加载缩略图

3. **数据存储**：
   ```dart
   // 新增字段
   exercise.thumbnail // 缩略图Base64
   exercise.boundingBox // 轨迹边界用于缩略图裁剪
   ```

#### 优点

- 实时反馈强，用户可见运动轨迹
- 交互自然，主流用户习惯
- 技术成熟，实现相对简单
- 无需API Key，无额外费用

#### 缺点

- 需要学习新库（flutter_map）
- 缩略图生成有性能开销
- OpenStreetMap国内访问可能不稳定
- 无离线地图功能

#### 技术方案

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 实时追踪 | 保持现有geolocator | 仅记录数据 |
| 详情地图 | `url_launcher`打开系统地图 | 无需集成SDK |
| 轨迹分享 | 生成KML/GPX文件 | 通用格式 |
| 缩略图 | 不生成 | 节省存储 |

#### 技术实现要点

1. **数据记录优化**：
   - 优化GPS采样频率（每3-5秒）
   - 添加轨迹平滑处理
   - 记录起点终点坐标

2. **系统地图唤起**：
   ```dart
   // 使用Google Maps URL Scheme
   final url = 'https://www.google.com/maps/dir/'
       '$startLat,$startLng/'
       '$endLat,$endLng';
   await launchUrl(Uri.parse(url));
   ```

3. **轨迹导出**：
   - 生成GPX格式文件
   - 支持分享到其他应用

#### 优点

- 无需地图SDK集成，开发量最小
- 系统地图体验原生
- 无存储缩略图开销
- 无需API Key费用

#### 缺点

- 实时运动时无地图反馈
- 用户无法在运动中看到轨迹
- 依赖系统地图应用（可能未安装）
- 无法显示实时配速

## 4. 方案对比

| 对比项 | 方案A（半屏实时地图） | 方案B（纯数据+系统地图） |
|--------|----------------------|--------------------------|
| **用户实时体验** | ★★★★★ 实时可见轨迹 | ★☆☆☆☆ 仅数字展示 |
| **实现复杂度** | ★★★☆☆ 中等 | ★★☆☆☆ 简单 |
| **开发工作量** | 约2-3天 | 约0.5-1天 |
| **依赖库** | flutter_map | url_launcher |
| **存储开销** | 需要存缩略图 | 无额外存储 |
| **离线支持** | 需额外处理 | 依赖系统地图 |
| **国内网络** | OSM可能不稳定 | 无影响 |

## 5. 推荐方案

### 首选方案A：半屏实时地图

理由：
1. 用户体验最佳，实时可见运动轨迹是核心需求
2. `flutter_map`是成熟开源方案，无需API Key
3. 符合主流健身APP交互习惯
4. 长期可扩展性强（热力图、离线地图）

### 实施建议

1. **第一阶段**（1-2天）：
   - 集成flutter_map
   - 实现实时地图展示
   - 保留现有数据面板

2. **第二阶段**（1天）：
   - 缩略图生成
   - 列表页地图卡片

3. **第三阶段**（可选）：
   - 轨迹渐变色（按速度/海拔）
   - 离线地图缓存
